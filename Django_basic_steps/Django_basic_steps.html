Django Basic Steps

Virtual Environment

  1. Create Virtual Environment

    cmd: ...path\Project>python –m venv virtual

    "virtual" is a name, It can be "anything"

  2. Activate Virtual Environment

    cmd: ...path\Project>virtual\Scripts\activate

Django Installation

  1. Install

    cmd: ...path\Project>pip install Django

  2. Verify Installation

    cmd: ...path\Project>django-admin --version

  3. Auto-generate requirements.txt

    (option1)cmd: ...path\Project>pip freeze > requirements.txt

    (option2)cmd: ...path\Project>pip install pipreqs
    ...path\Project>pipreqs /path/to/project

  4. Auto-install from requirements.txt

    cmd: ...path\Project>pip install -r requirements.txt

Django Create Project

  1. Create Project's Container

    cmd: ...path\Project>django-admin startproject contName

    "contName"(containerName) can be anything, inside it are the project files

  2. Check Temporary Server

    cmd: ...path\Project\contName>py manage.py runserver

  3. Create App

    cmd: ...path\Project\contName>py manage.py startapp appName

    "appName" can be anything. An app can be in multiple projects.

  4. Write View.py for test

    Edit view.py guidelines

    This is optional in order to display something on the browser

    Guidelines
    Inform Django - Models Create/Change

    cmd: ...path\Project\contName>py manage.py makemigrations appName
    The same applies when Models do not exist yet

    Preview Database Details

    cmd: ...path\Project\contName>py manage.py sqlmigrate appName fileNumber
    "fileNumber" is the number in the database filename, e.g. appName\migration\0001_initial.py

    Check for errors

    cmd: ...path\Project\contName>python manage.py check

    Apply Model Changes to Database

    cmd: ...path\Project\contName>python manage.py migrate

    Three step guide
    • Change your models (in models.py).
    • Run python manage.py makemigrations to create migrations for those changes
    • Run python manage.py migrate to apply those changes to the database

    The reason that there are separate commands to make and apply migrations is because you’ll commit migrations to your version control system and ship them with your app

Django Admin

  1. Philosophy

    - Generating admin sites for your staff or clients to add, change, and delete content is tedious work that doesn’t require much creativity. For that reason, Django entirely automates creation of admin interfaces for models.
    - Django was written in a newsroom environment, with a very clear separation between “content publishers” and the “public” site. Site managers use the system to add news stories, events, sports scores, etc., and that content is displayed on the public site. Django solves the problem of creating a unified interface for site administrators to edit content.
    - The admin isn’t intended to be used by site visitors. It’s for site managers.

  2. Create Admin user

    cmd: ...path\Project\contName>py manage.py createsuperuser

    Then Set Username, Email address & Password, they will automatically be asked

  3. Display Admin Page

    If development server is not running:

    cmd: ...path\Project\contName>py manage.py runserver

  4. Make the "App" modifiable by Admin

    .../app/admin.py(write code):
    from django.contrib import admin
    from .models import Question
    admin.site.register(Question)
    "Question" is an example of a Model

    To be able to edit a Model(eg. Question), it must be registered here

  5. Edit Models

    The data of the Models (ig. field values) can be altered,
    but the structure of the Models (ig. field attributes) cannot
